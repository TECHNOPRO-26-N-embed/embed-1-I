### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（リモコン操作で音選択）──→ [音再生中] ──（停止コマンド/曲終了）──→ [待機中]
    │                                              ↑
    ├─（曲作成モード選択）──→ [曲作成中] ──（保存/終了）──→ [待機中]
    └─（目覚ましモード選択）→ [目覚まし待機] ──（アラーム発動）──→ [音再生中]
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| （例）待機中 | センサー値を100msごとに監視する | 緑LED：500msで点滅 |
| 待機中         | リモコン入力待ち・LCDに待機表示              | LED消灯、LCDに待機メッセージ             |
| 音再生中       | 選択した音を再生                            | パッシブブザーで音、LEDが音階に応じ点灯   |
| 曲作成中       | リモコンや入力で音階・長さを記録            | 入力に応じてLED点灯、LCDに作成情報表示   |
| 目覚まし待機   | 設定時刻まで待機                            | LED消灯または点滅、LCDに時刻表示         |
| アラーム発動   | 目覚まし時刻到達                            | ブザー・LED・モーター動作                |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| （例）LED点滅 | 500ms | **millis** | ボタン監視と並行するため |
| （例）センサー読み取り | 100ms | **millis** | 常時監視が必要なため |
| 処理           | 必要な周期 | delay / millis | 判断の理由                       |
|:--|:--|:--|:--|
| LED点滅        | 500ms      | millis         | 他の入力監視と並行               |
| ボタン監視     | 50ms       | millis         | チャタリング対策                 |
| リモコン監視   | 100ms      | millis         | 常時監視                         |
| 曲再生         | 音長ごと   | millis         | 他の処理と並行                   |
| 目覚まし監視   | 1s         | millis         | 秒単位で十分                     |
|  |  |  |  |
|  |  |  |  |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース         | 発生条件                       | 検知方法                                         | 対応方針                                  |
|:--|:--|:--|:--|

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」


**AIの回答（要約）：**
・ピン競合なし（D0/D1, A4/A5, D10-D13の用途を明記し、競合回避済み）
・主要ライブラリ（IRremote, LiquidCrystal_I2C等）は競合なし
・状態遷移は全状態から待機に戻れる設計で抜けなし
・delay()は使わず、全てmillis()管理で並行処理・応答性を確保

**対応した内容：**
・ピン割当・競合を再確認し、競合なしを明記
・状態遷移図・意味表で全状態の遷移ルートを明示
・delay()を使わずmillis()でタイミング管理する設計に統一

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」


**AIの回答（要約）：**
・チャタリング対策はreadButton()等で50msデバウンス処理を実装予定
・delay()は使わず、全てmillis()で管理しているため応答性に問題なし
・アナログセンサーは閾値バッファでノイズ対策を設計済み

**対応した内容：**
・ボタン/リモコン入力はデバウンス処理を明記
・delay()を使わない設計を徹底
・アナログ入力はバッファ処理でノイズ対策を記載
